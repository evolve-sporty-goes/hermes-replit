# Workspace-to-Remote Commit Workflow

When working in a separate workspace directory (not the original repo checkout), the commit/push workflow has several pitfalls that differ from normal feature-branch work.

## Scenario

You have a workspace directory (e.g., `/home/runner/workspace`) that is either:
- A clone of the original repo
- A directory that was copied/synced from another location
- Contains large runtime state (databases, caches, LSP binaries) that must NOT be committed

## Full Workflow

```bash
# 1. Check status
cd /home/runner/workspace
git status

# 2. Review diffs before staging
git diff --stat          # see what changed
git diff <file>          # inspect specific files

# 3. Stage changes
git add <files>          # stage intentionally
# OR
git add .                # stage everything (review first!)

# 4. Handle deleted files properly
git rm --cached <deleted_file>   # if git shows "deleted" but you didn't git rm

# 5. Commit
git commit -m "type(scope): description"

# 6. Push — handle divergence
git push origin main
# If rejected:
git stash                # if unstaged changes block rebase
git pull --rebase origin main
git stash pop
git push origin main

# 7. After push — handle transient files
# SQLite WAL/SHM files may appear as unstaged after operations:
#   .hermes_data/state.db-shm
#   .hermes_data/state.db-wal
# These are runtime state — add to .gitignore or leave unstaged.
```

## Key Pitfalls

### Transient files appearing after commit

After `git stash pop` or after committing, runtime files like SQLite WAL/SHM may appear as unstaged changes. These are regenerated by the application at runtime and should never be committed.

**Fix:** Add patterns to `.gitignore` before the first commit:

```gitignore
# SQLite transient files
*.db-shm
*.db-wal

# LSP binaries (if vendored)
lsp/bin/
```

### Large binary/state files

If your workspace contains large files (>10MB) like `state.db`, `models_dev_cache.json`, or language-server binaries, consider whether they belong in git at all. GitHub has a 100MB per-file limit and large files bloat the repo forever.

**Options:**
- Add to `.gitignore` if they're runtime/regenerable
- Use `git lfs` if they must be versioned
- Use a separate backup mechanism (rsync, S3) for state

### Remote divergence from concurrent pushes

If CI or another agent pushes to the same branch between your `git add` and `git push`, the push will be rejected. Always:

1. `git pull --rebase` before pushing
2. Rebase (never force-push) to integrate remote changes
3. Push immediately after rebase

### Credential safety

When committing a workspace that contains secrets (`.env`, `auth.json`):
- Ensure `.env` is in `.gitignore` (never commit secrets)
- If secrets must be in the repo (e.g., for deployment), use encrypted secrets (`git-crypt`, `sops`)
- After pushing, verify no secrets leaked: `git log --all --full-history -- .env`
